# ADR-010: buf.build for Protobuf Management

**Status:** Accepted
**Date:** 2026-03-18

## Context

jitsudo uses protobuf as the API contract for both gRPC and REST (via grpc-gateway). Managing protobuf tooling historically requires installing `protoc`, managing multiple `protoc` plugins, pinning plugin versions, and writing complex `Makefile` rules. This is a common source of contributor friction ("it generates differently on my machine").

## Decision

[buf.build](https://buf.build) manages all protobuf tooling: linting, breaking change detection, and code generation.

- `buf.yaml` defines the module and lint rules
- `buf.gen.yaml` defines code generation (Go stubs, grpc-gateway, OpenAPI v2)
- `make proto` runs `buf generate` — a single command produces all generated Go code
- CI runs `buf lint` to enforce protobuf style rules
- CI runs `buf breaking` on PRs to detect breaking API changes

## Consequences

**Positive:**
- `buf generate` produces deterministic output regardless of the local OS or machine — eliminates "different output on different machines"
- `buf lint` enforces protobuf style rules (naming conventions, package versioning) automatically
- `buf breaking` catches breaking API changes before they merge — prevents accidental incompatibilities between CLI and server versions
- Remote plugins (via `buf.build/...`) mean contributors don't install `protoc` or manage plugin binaries locally; `buf` fetches and caches them automatically
- The buf.build registry hosts well-known proto dependencies (googleapis, grpc-gateway) — no manual `$GOPATH/pkg/mod` proto file hunting

**Negative:**
- `buf` is a required build tool (in addition to Go). It must be installed by contributors (`brew install bufbuild/buf/buf` on macOS).
- Remote plugin execution requires internet access during code generation (mitigated by buf's local cache)
- Generated files are `.gitignore`'d and regenerated by each developer, which means PRs that change `.proto` files must also run `make proto` and commit the results

**Generated file strategy:**
Generated Go files (in `internal/gen/`) are `.gitignore`'d and not committed to the repository. Every developer runs `make proto` after cloning. CI verifies that `buf generate` produces no diff against the checked-in generated files (TODO: add this check).
