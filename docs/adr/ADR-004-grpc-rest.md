# ADR-004: gRPC + REST Dual API Surface

**Status:** Accepted
**Date:** 2026-03-18

## Context

jitsudo's control plane needs to expose an API for:
- The CLI (`jitsudo` binary)
- Future web UI
- Approval bots and external integrations
- The public Go client library (`pkg/client`)

The API must be versioned, well-documented, and accessible to clients that may not speak gRPC (e.g., curl, webhook receivers, browser-based approval UIs).

Options considered: REST-only, gRPC-only, gRPC + REST transcoding via grpc-gateway.

## Decision

The API is defined in protobuf (managed by buf.build) and exposed as both:
- A native gRPC API
- A REST/JSON API via grpc-gateway (REST transcoding)

Both surfaces are generated from the same protobuf source — no duplicate implementation.

## Consequences

**Positive:**
- Single source of truth: the `.proto` files define the API contract; both gRPC and REST are derived from it
- Strong typing and versioning at the protobuf layer catch breaking changes before they reach clients
- The OpenAPI v2 spec generated by `buf.gen.yaml` enables API documentation, SDK generation, and `curl`-based exploration
- gRPC provides efficient binary encoding and streaming for future use cases (e.g., real-time status updates)
- REST makes the API accessible to any HTTP client, including approval bots using webhooks and browser-based UIs

**Negative:**
- grpc-gateway adds complexity vs. a pure REST implementation
- buf.build and the protobuf toolchain are additional developer dependencies
- REST URL path conventions are constrained by how grpc-gateway maps gRPC method names

**Why not REST-only:**
REST-only was considered as the simpler starting point. However, adding gRPC after establishing a REST API typically requires a significant refactor or maintaining two implementations. Starting from protobuf ensures the API shape is explicit and versioned from day one, and gRPC enables efficient CLI-to-server communication.

**Why not gRPC-only:**
gRPC-only would exclude all non-gRPC clients (curl, browser UIs, simple webhook integrations). grpc-gateway is a mature, well-maintained solution for this problem with minimal overhead.
